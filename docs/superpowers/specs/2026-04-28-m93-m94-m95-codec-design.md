# M93 / M94 / M95 — Codec Parity Design

**Status:** brainstorm complete; awaiting user spec review.
**Date:** 2026-04-28.
**Drives:** `WORKPLAN.md` Phase 7 (M92 release prep). The v1.2.0 acceptance gate "TTI-O within 15% of CRAM 3.1 lossless" requires three new codecs that close the architectural gap with CRAM 3.1's reference-based + fqzcomp pipeline.

---

## 1. Problem statement

The M92 smoke benchmarks on chr22 (`docs/benchmarks/v1.2.0-smoke-report.md`) show TTI-O at **2.50× CRAM 3.1** on real NA12878 chr22 (1.78M reads, aux-stripped) — far over the 1.15× acceptance gate. Decomposition shows the gap is dominated by two architectural items already deferred in `WORKPLAN.md` "Deferred to v1.1+":

| Channel | TTI-O current | CRAM 3.1 | Gap |
|---|---|---|---|
| `sequences` | 45 MB (`BASE_PACK`, reference-free 2-bit) | ~5 MB (reference-aligned diff) | **~40 MB** |
| `qualities` | 110 MB (`RANS_ORDER0`) | ~55 MB (fqzcomp / rANS-Nx16) | **~55 MB** |
| integer channels (`positions`, `flags`, `mate_info_pos`, etc.) | ~17 MB (gzip) | ~10 MB (delta + entropy) | **~7 MB** |
| HDF5 structural | ~12 MB (default 1024-row chunks → ~21K chunks total) | ~2 MB | **~10 MB** |

Closing all four gaps requires a coordinated set of three milestones (M93, M94, M95) before M92 release prep can resume.

## 2. Decisions captured during brainstorming

| Q | Decision |
|---|---|
| Q1 — Reference storage strategy | **C** — hybrid covered-span: embedded by default (covered chromosomes only); opt-out to external-by-MD5 via `embed_reference=False`. |
| Q2 — Codec interface | **A** — extend per-channel codec model with a new `is_context_aware: bool` declaration. Context-aware codecs receive sibling channels + reference resolver at write/read time via a new M86 pipeline hook. No `RunCodec` super-abstraction. |
| Q3a — Slice strategy for M93 | **B** — fixed-read-count slices, 10K reads/slice (CRAM-aligned). Slice index in codec wire format. |
| Q3b — Compression target | Push for **≤1.15× CRAM 3.1**; M93 + M94 + M95 sized to hit it. |
| Q4 — fqzcomp variant | **C** — fqzcomp-Nx16 (Bonfield 2022, CRAM 3.1 default). Interleaved 4-way rANS for SIMD parallelism. |
| Q5a — Defaults | **B without feature flag** — new codec stack is the v1.2.0 default. v1.1.x readers reject v1.2 files via existing M82 schema gate; no dual-read shim. |
| Q5b — REF_DIFF write fallback (no ref provided) | **C** — per-channel `@compression` reflects what was actually applied; sequences fall back to `BASE_PACK` (id 6); other channels still get their codecs. |
| Q5c — REF_DIFF read fallback (ref unresolvable) | **A** — hard error (`RefMissingError`); no partial decode. |
| Q6 — Reference scope inside the file | **C** — auto-deduplicate: per-run by default; if another run in the same file already wrote `/study/references/<reference_uri>/`, link rather than duplicate. |

## 3. Milestone breakdown

### M93 — REF_DIFF codec (~3 weeks)

Reference-based sequence diff codec. Replaces `BASE_PACK` as the default for `signal_channels/sequences` when a reference is available.

**Wire format (codec id 9):**

```
[Codec header, 38 + N bytes total]
  magic                : 4 bytes  "RDIF"
  version              : uint8    = 1
  reserved             : uint8[3] = 0
  num_slices           : uint32
  total_reads          : uint64
  reference_md5        : 16 bytes
  reference_uri_len    : uint16
  reference_uri        : N bytes  (UTF-8, length = reference_uri_len)
  // header_size = 4 + 1 + 3 + 4 + 8 + 16 + 2 + N = 38 + N

[Slice index, num_slices × 32 bytes]
  for each slice:
    body_offset      : uint64    (relative to slice bodies block)
    body_length      : uint32
    first_position   : int64
    last_position    : int64
    num_reads        : uint32
  // entry_size = 8 + 4 + 8 + 8 + 4 = 32

[Slice bodies, concatenated]
  for each slice (independently rANS-encoded):
    For each read in slice:
      for each cigar M-op base:
        1 bit: match-ref (0) vs substitution (1)
        if substitution: 8 bits: actual base
      for each cigar I-op or S-op:
        op_length × 8 bits: inserted/soft-clipped bases
      // D, H, N, P ops: no payload (lengths come from cigar channel)
```

The bit-packed M-op flag stream is the load-bearing trick: at >99% sequencing accuracy, ~99% of M-op bits are `0` (match), which rANS compresses heavily by entropy.

**Algorithm — encode:**

1. Walk reads in coordinate order; partition into 10K-read slices.
2. For each slice, for each read: walk the cigar ops in lock-step against the reference. Emit a flag per M-op base, append substitution bytes when flag=1, append all I/S-op bases verbatim.
3. rANS-encode the slice's flag-and-substitution bitstream as a single byte stream. Cigar D/H/N/P ops carry no payload (their lengths come from the `cigars` channel).
4. Build slice index: each slice records its byte offset, length, first/last reference position, read count.
5. Write wire format: codec header + slice index + concatenated slice bodies.

**Algorithm — decode:**

1. Parse codec header. Resolve reference via the lookup chain: embedded `/study/references/<reference_uri>/` (verify MD5) → external `REF_PATH` env var or explicit `external_reference=` arg → `RefMissingError`.
2. Locate slices overlapping the query region (or all slices for full-channel reads).
3. For each requested slice: rANS-decode body. Walk reads in order, replaying cigar ops against the reference, applying flags + substitution bytes, emitting reconstructed sequences.

**Reference storage in HDF5:**

```
/study/references/<reference_uri>/
  @md5             = "a718acaa6135fdca8357d5bfe94211dd"
  @reference_uri   = "GRCh37.hs37d5"
  chromosomes/
    22/
      @length      = 51304566
      data         : zlib-compressed uppercase ACGTN bytes
    21/
      ...
```

Auto-deduplication (Q6): when writing run B against `reference_uri = X`, the encoder checks whether `/study/references/X/` already exists in this file. If yes, only the run's `@reference_uri` attribute is written. If no, the chromosomes touched by the union of runs sharing `X` are embedded.

**Edge cases:**

- Empty slice (no reads): not allowed at write; encoder errors. Decoder is robust to slices with `num_reads = 0` but encoder doesn't emit them.
- Read with cigar `*` (unmapped): NOT processed by REF_DIFF. The pipeline routes unmapped reads through `BASE_PACK` (id 6) on a separate sub-channel. **Open question:** how exactly to split mapped vs unmapped at the channel level? Most likely: introduce a `sequences_unmapped` virtual channel for unmapped reads (parallel to the existing `mate_info_*` virtual channels). Resolved during M93 implementation.
- Read with cigar containing `=` (explicit match) / `X` (explicit mismatch): encoder treats both as M (no flag distinction) since the M-op bit-flag already captures match/mismatch.
- Reference too long to embed: encoder errors when total ref bytes > `EMBED_REF_THRESHOLD_MB` (default 1024 MB). User must pass `embed_reference=False` and provide external ref at decode.

### M94 — FQZCOMP_NX16 codec (~3 weeks)

fqzcomp-Nx16 lossless quality compressor (Bonfield 2022 / CRAM 3.1 default).

**Wire format (codec id 10):**

```
[Codec header, 50 + L bytes total]
  magic                : 4 bytes  "FQZN"
  version              : uint8    = 1
  flags                : uint8    (see Flags byte layout below)
  num_qualities        : uint64   (decoded length, total bytes)
  num_reads            : uint32   (for read-boundary marking)
  rlt_compressed_len   : uint32   (length of read_length_table block in bytes)
  read_length_table    : L bytes  (uint32[num_reads], rANS_ORDER0-compressed; L = rlt_compressed_len)
  context_model_params : 16 bytes (see Context model params layout below)
  state_init[4]        : 4 × uint32  (initial rANS states; little-endian)

[Frame, 4-way interleaved rANS body]
  N bytes — output bytes from the four parallel rANS encoders, written
  round-robin: byte 0 from state[0], byte 1 from state[1], ... byte 3
  from state[3], byte 4 from state[0], etc. N = ceil(num_qualities / 4)
  × 4 (each state emits exactly num_qualities/4 bytes, last state may
  emit one extra; padding with zero bytes if num_qualities not a
  multiple of 4 — recorded in the flags byte).

[Trailer, 16 bytes]
  state_final[4]       : 4 × uint32  (final rANS states; little-endian)
```

**Flags byte layout:**

```
bit 0: has_revcomp_context  (1 = read flag bit 16 contributes to context)
bit 1: has_pos_context      (1 = position-within-read contributes)
bit 2: has_length_context   (1 = read-length bucket contributes)
bit 3: has_prev_q_context   (1 = up to 3 previous quality bytes contribute)
bit 4-5: padding_count      (0..3 zero bytes appended to last 4-way row)
bit 6-7: reserved           (must be 0 in v1)
```

**Context model params layout (16 bytes):**

```
context_table_size_log2 : uint8   (0..16, table has 2^N entries; default 12 = 4096 contexts)
learning_rate           : uint8   (default 16)
max_count               : uint16  (default 4096; renormalisation triggers when any freq > this)
freq_table_init         : uint8   (0 = uniform, 1 = empirical-prior; default 0)
context_hash_seed       : uint32  (LE; default 0xC0FFEE)
reserved                : uint8[7] (must be 0)
```

The four `state_init` and `state_final` values are the lifecycle markers for the 4-way rANS state machine; they must match exactly between encoder and decoder.

**Context model:**

For each quality byte, the context for predicting it is the tuple:

```
(prev_q[0], prev_q[1], prev_q[2], position_bucket, revcomp_flag, length_bucket)
```

Where:
- `prev_q[0..2]` — three immediately preceding quality bytes (use `0` if at start of read).
- `position_bucket` — 4-bit bucketed position-within-read (0..15 spanning 0..read_length).
- `revcomp_flag` — 1 if read flag has reverse-complement bit set, else 0.
- `length_bucket` — 3-bit bucket of read length (0..7 covering common Illumina/PacBio read lengths).

Context vector compresses to a hashed `uint16` index into a context table. Each context maintains a 256-entry symbol-frequency table.

**Adaptive frequency table updates (deterministic, identical across implementations):**

```
// After encoding/decoding symbol s in context c:
c.freq[s] += LEARNING_RATE   // = 16
if max(c.freq) > MAX_COUNT:  // = 4096
    for each symbol s' in 0..255:
        c.freq[s'] = max(1, c.freq[s'] >> 1)
```

The renormalisation schedule is the cross-language byte-exact pinch point. Conformance fixtures explicitly target the renormalisation boundary (frequency 4096 → 2048 transition).

**Interleaved 4-way rANS:**

Quality byte stream split into 4 substreams: byte 0, 4, 8, ... → stream 0; byte 1, 5, 9, ... → stream 1; etc. Each substream encodes with its own rANS state. Output bytes interleaved round-robin.

State arithmetic: 32-bit unsigned. Java implementation uses `long` with `& 0xFFFFFFFFL` after each operation.

**Python implementation strategy:**

Pure Python is impractical (~30 min per chr22 fixture vs ~5s native). Python ships a C extension at `python/src/ttio/codecs/_fqzcomp_nx16/`:

- `_fqzcomp_nx16.c` — ~500 lines, the inner state machine.
- `_fqzcomp_nx16_wrapper.py` — ~200 lines, ctypes/cffi shim.
- Build via `setuptools` `Extension`. Adds `Cython>=3.0` to build deps in `pyproject.toml`.

ObjC and Java implementations are native (no extension required).

### M95 — Integer + structural compression (~1.5 weeks)

**New codec id 11 — `DELTA_RANS_ORDER0`:**

```
[Codec header, 8 bytes]
  magic       : 4 bytes  "DRA0"
  version     : uint8    = 1
  element_size: uint8    (1 = int8, 4 = int32, 8 = int64)
  reserved    : uint8[2] = 0

[Body, RANS_ORDER0-encoded byte stream]
  zigzag-varint deltas of consecutive elements (output[i] - output[i-1]),
  with output[-1] = 0 implicit
```

**Channel-codec assignments (M95 defaults):**

| Channel | Codec | Notes |
|---|---|---|
| `positions` (int64, sorted ascending) | `DELTA_RANS_ORDER0` (id 11) | Deltas typical 100–500 at 30× cov. |
| `flags` (uint32) | `RANS_ORDER0` (id 4) | LE bytes; ~5 dominant flag values. |
| `mapping_qualities` (uint8) | `RANS_ORDER0` (id 4) | Discrete BWA mapq distribution. |
| `template_lengths` (int32) | `RANS_ORDER0` (id 4) | Bimodal at ±tlen. |
| `mate_info_pos` (int64) | `RANS_ORDER0` (id 4) | Non-monotonic; delta unhelpful. |
| `cigars` | `RANS_ORDER1` (id 5) | UNCHANGED — already strong. |
| `read_names` | `NAME_TOKENIZED` (id 8) | UNCHANGED. |
| `mate_info_chrom` | `NAME_TOKENIZED` (id 8) | UNCHANGED. |

**HDF5 chunk-size tuning:**

`WrittenGenomicRun.chunk_size_hint: int = 65536` (raised from default 1024). Applies to per-channel datasets and variable-length compound channels (`cigars`, `read_names`, `mate_info_chrom`).

For 1.78M reads × 12 channels: ~330 chunks total at 65K-row chunks vs ~21K chunks at 1024-row. Net structural savings: ~3 MB.

## 4. Format-version bump

`@ttio_format_version` bumps **1.4 → 1.5**. v1.2 writes always emit 1.5. v1.1.x and earlier readers reject the file at format-version check (existing M82 schema gate; no new logic).

v1.2 readers accept format versions 1.1–1.5 for backwards compatibility — codec dispatch is per-channel `@compression`, so old files with codecs 0–8 still decode through their existing code paths. Only files with codec ids 9/10/11 require v1.5 reader support.

## 5. M86 pipeline integration

The codec dispatch logic in `SpectralDataset._write_genomic_run_subtree` (Python ref, mirrored in ObjC + Java) gains a context-aware hook:

```python
def _dispatch_codec(channel_name, channel_data, codec_id, context):
    """context: optional dict carrying positions, cigars, ref_resolver
       for context-aware codecs."""
    codec = get_codec(codec_id)
    if codec.is_context_aware:
        return codec.encode(channel_data, **context.required_keys)
    else:
        return codec.encode(channel_data)
```

`Codec.is_context_aware: bool` is a class-level declaration. Only `REF_DIFF` is context-aware in v1.2; the hook is generic for future codecs.

## 6. Default codec selection (Q5a, no feature flag)

`signal_compression="gzip"` (the default) becomes a marker meaning "use the v1.5 default stack":

```python
DEFAULT_CODECS_V1_5 = {
    "sequences":          Compression.REF_DIFF,         # falls back to BASE_PACK if no ref
    "qualities":          Compression.FQZCOMP_NX16,
    "positions":          Compression.DELTA_RANS_ORDER0,
    "flags":              Compression.RANS_ORDER0,
    "mapping_qualities":  Compression.RANS_ORDER0,
    "template_lengths":   Compression.RANS_ORDER0,
    "mate_info_pos":      Compression.RANS_ORDER0,
    "cigars":             Compression.RANS_ORDER1,
    "read_names":         Compression.NAME_TOKENIZED,
    "mate_info_chrom":    Compression.NAME_TOKENIZED,
}
```

`signal_compression="none"` keeps current "no codec, raw byte storage" behavior — mostly for debugging.

## 12. New binding decisions (proposed)

| # | Decision | Rationale |
|---|---|---|
| 80a | REF_DIFF wire format uses bit-packed M-op flags + raw I/S-op bases; D/H/N/P ops carry no payload. | Cigar channel already encodes structure; redundant payload would inflate. |
| 80b | Embedded references stored at `/study/references/<reference_uri>/`; auto-deduplicated within a single file by URI. | Multi-omics files (M91) frequently share a reference; per-run duplication wastes space. |
| 80c | REF_DIFF read-time fallback: hard error when reference not resolvable. No partial decode. | Genomic data integrity is non-negotiable; partial decode is worse than failure. |
| 80d | FQZCOMP_NX16 context model uses (prev_q[0..2], position_bucket, revcomp_flag, length_bucket); deterministic adaptive frequency table renormalisation at 4096-count boundary, halve-with-floor-1. | Matches CRAM 3.1 default fqzcomp parameters; deterministic across implementations. |
| 80e | FQZCOMP_NX16 Python implementation links to a C extension; ObjC + Java implementations are native. | Pure Python impractical at chr22-scale (~30 min vs ~5s); accepted build-deps cost. |
| 80f | DELTA_RANS_ORDER0 wire format includes a 1-byte `element_size` field (1 / 4 / 8). | Decoder needs to know the natural dtype to reverse zigzag-varint decode. |
| 80g | v1.5 format-version is a clean break — no feature flag, no dual-read shim. v1.1.x readers reject via existing schema check. | Per Q5a; aligns with Binding Decision §74 (clean-break philosophy from M80 rebrand). |

## 14. Open questions (resolved during implementation, not blocking spec)

- M93: how to handle unmapped reads. Most likely a `sequences_unmapped` virtual channel routed through `BASE_PACK` (id 6).
- M94: exact `length_bucket` boundaries. Set during M94 prototyping based on observed read-length distributions in the chr22 / synthetic fixtures.
- M95: whether `mate_info_pos` benefits from a "delta-from-current-read-position" transform. Try both during M95; pick the better empirically.
- HDF5 chunk size 65536 may be too aggressive for small runs (e.g. M91 metabolomics with 100 spectra). Add a runtime check: if `num_reads < 4 × chunk_size_hint`, lower to `max(1024, num_reads // 4)`.
