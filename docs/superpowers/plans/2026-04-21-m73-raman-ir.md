# M73 — Raman / IR Spectrum Support

**Goal:** Extend the Spectrum hierarchy with vibrational-spectroscopy
modalities (Raman and mid-IR), reaching the same three-language parity
contract as MS and NMR. Four new leaf classes plus one importer +
exporter pair.

**License:** LGPL-3.0 (data classes, HDF5 I/O), Apache-2.0 (importer,
exporter).

**Batched under:** v0.11.0 (first of five queued milestones; see
HANDOFF.md "Deferred to v1.0+" queue).

## Design

### Four new classes

| 1-D leaf | 2-D leaf | Inherits from | Analogue |
|---|---|---|---|
| `RamanSpectrum` | `RamanImage` | `Spectrum` / composition-on-`SpectralDataset` | `NMRSpectrum` / `MSImage` |
| `IRSpectrum` | `IRImage` | same | same |

The 1-D classes follow the `NMRSpectrum` / `MassSpectrum` template
exactly: a pair of `SignalArray`s (x-axis + intensity) plus a few
modality-specific scalars, inheriting from `Spectrum`.

The 2-D classes follow `MSImage` exactly: a `height × width ×
spectral_points` intensity cube with spatial `pixel_size_x` /
`pixel_size_y` and `scan_pattern`, composing dataset-level metadata
the way `MSImage` does.

### Why 2-D = imaging/mapping, not 2-D correlation spectroscopy

The two natural interpretations of "2-D Raman/IR":

1. **Imaging / mapping** — a spatial grid of 1-D spectra (Raman
   mapping, FTIR microscopy). Analogous to `MSImage`.
2. **2-D correlation spectroscopy (2D-COS)** — a synchronous /
   asynchronous intensity matrix with two spectral axes. Analogous
   to `NMR2DSpectrum`.

This plan picks **(1) imaging/mapping** as the 2-D interpretation.
Rationale: Raman mapping and FTIR microscopy are the dominant
real-world "2-D" use cases in vibrational spectroscopy.

**Confirmed 2026-04-21:** (1) for M73; 2D-COS deferred as a future
option (`Raman2DCorrelation` / `IR2DCorrelation` leaves on the
`NMR2DSpectrum` template, separate milestone).

### Modality-specific fields

| Class | Extra fields (beyond `Spectrum`) |
|---|---|
| `RamanSpectrum` | `wavenumberArray` (cm⁻¹), `intensityArray`, `excitationWavelengthNm` (double), `laserPowerMw` (double, optional), `integrationTimeSec` (double) |
| `IRSpectrum` | `wavenumberArray` (cm⁻¹), `transmittanceOrAbsorbanceArray`, `mode` (enum: `transmittance` / `absorbance`), `resolutionCmInv` (double), `numberOfScans` (int, optional) |
| `RamanImage` | `intensityCube` rank-3, `wavenumbers` `SignalArray`, `pixelSizeXMicrons`, `pixelSizeYMicrons`, `scanPattern`, `excitationWavelengthNm`, `laserPowerMw` |
| `IRImage` | `intensityCube` rank-3, `wavenumbers` `SignalArray`, `pixelSizeXMicrons`, `pixelSizeYMicrons`, `scanPattern`, `mode` (transmittance/absorbance), `resolutionCmInv` |

Units follow the dominant literature convention — cm⁻¹ for
vibrational frequency, micrometers for pixel pitch (IR/Raman
microscopy typical scale). `AxisDescriptor` carries the full
ontology-backed unit via `CVParam`; these scalars are convenience.

### HDF5 layout

Reuses the existing `Spectrum` / `MSImage` layout with modality
discriminator stored as a string attribute on the group:

- 1-D: `@spectrum_type = "raman"` or `"ir"`; signal channels named
  `wavenumber` + `intensity` (or `transmittance`/`absorbance`);
  modality scalars on the group.
- 2-D: `@dataset_type = "raman_image"` or `"ir_image"`; cube in
  `intensity` dataset, `wavenumbers` in a 1-D dataset on the same
  group, spatial + modality scalars on the group.

No format-spec bump required — `spectrum_type` / `dataset_type`
already discriminate.

### Importer / exporter: JCAMP-DX

**Choice:** JCAMP-DX 5.01 (IUPAC standard). `.jdx` / `.dx` extension.

Why:
- Single format covers both Raman and IR, plus UV-Vis, NIR, MS, NMR
  as a bonus (we only wire the vibrational subset).
- ASCII, no vendor-specific binary quirks — a clean first target.
- The spec is open and stable; test fixtures are widely available
  (JCAMP-DX test suite, public databases).

**Scope for M73 (confirmed 2026-04-21):** 1-D `RamanSpectrum` +
`IRSpectrum` only. 2-D imaging cubes (RamanImage / IRImage) are 10–
100 MB as plain ASCII at realistic grid sizes — impractical.
`NTUPLES` / 2-D JCAMP-DX is deferred to M73.1 alongside the binary
vendor formats. HDF5 remains the canonical 2-D round-trip for M73.

Deferred to follow-on milestones (per user direction):
- Thermo / Nicolet SPA (binary FTIR)
- Thermo Galactic SPC
- Renishaw WDF (Raman imaging binary)
- Bruker OPUS

## Cross-language parity contract

Three-way conformance test `test_raman_ir_cross_language.py` (Python-
driven):

1. Python writes a synthetic Raman 1-D + Raman image + IR 1-D + IR
   image `.mpgo` file.
2. Spawns `MpgoVerify` (ObjC) and the Java `MpgoVerify` CLI on the
   file; diffs JSON summaries.
3. JCAMP-DX: Python writes → ObjC reads → Java reads, diff all three
   in-memory representations to bit-identical intensity arrays.

Passes iff all three implementations produce byte-identical HDF5
groups, equal-ish (within 1 ULP) intensities after JCAMP-DX
round-trip, and matching class-identification strings in
`MpgoVerify` JSON.

## Open decisions to confirm before implementation

1. **2-D = imaging/mapping (this plan) vs 2-D correlation
   spectroscopy.** **Confirmed (1) 2026-04-21.** 2D-COS deferred.
2. **Intensity-cube dtype default.** `MSImage` uses `float64`.
   Raman imaging is commonly int16/int32 CCD counts — preserve
   `float64` default for parity, or allow the 2-D classes to
   declare dtype via `EncodingSpec` at construction time? Plan
   picks the latter (more flexible, no performance cost).
3. **JCAMP-DX dialect coverage.** Full 5.01 feature support is
   large; scope to vibrational `XYDATA` + `XYPOINTS` + `NTUPLES`
   blocks, skip LINK/MS/NMR sub-types, skip audit-trail
   annotations. Document the excluded surface in
   `vendor-formats.md`.
4. **Filename prefix for Python modules.** `raman_spectrum.py` /
   `ir_spectrum.py` mirrors the existing `mass_spectrum.py` /
   `nmr_spectrum.py` convention. Confirmed implicitly.
