# Architectural primitives for a unified multi-omics data standard

Mass spectrometry and NMR spectroscopy share more architectural DNA than their fragmented format ecosystems suggest. An analysis of 15+ data formats across both domains — from mzML and nmrML through vendor binaries to modern HDF5 and columnar stores — reveals a convergent set of **six fundamental data primitives** and **three container patterns** that could serve as the foundation for a unified "TTI-O" standardization effort. The most critical finding: the `SignalArray` (a typed numeric buffer with axis metadata, encoding descriptor, and controlled vocabulary annotations) is the universal atom of information across every format examined. Where formats diverge is not in what they represent but in *how they package it* — XML vs. HDF5 vs. SQLite vs. directory trees — and in what they omit: none provide encryption, streaming transport, or standardized compressed-domain query APIs. MPEG-G (ISO/IEC 23092) for genomics has already solved these exact problems with its hierarchical containers, descriptor stream separation, multi-level protection, and normative APIs. The architectural mapping from MPEG-G concepts to MS/NMR needs is remarkably clean, suggesting that a formal ISO-track multi-omics standard is both feasible and overdue.

## Six data primitives are shared across every format

The most consequential finding from analyzing mzML, nmrML, mzMLb, imzML, mzIdentML, mzTab-M, TraML, NMR-STAR, NMReDATA, Bruker TDF, Thermo RAW, and a dozen other formats is that they all encode combinations of just six fundamental primitives. Understanding these primitives is the first step toward abstraction.

**Primitive 1: The SignalArray.** Every format stores typed numeric buffers paired with axis descriptors. In mzML, this is a `<binaryDataArray>` containing base64-encoded float32/float64 values annotated with cvParams specifying the axis (m/z, intensity, retention time). In nmrML, the same pattern appears as `<fidData>` — a base64-encoded complex128 buffer (interleaved real+imaginary pairs) with a time axis. In Bruker's NMR `fid` files, it is raw 32-bit integers with byte order specified in the JCAMP-DX `acqus` parameter file. In mzMLb, it becomes native HDF5 datasets named by CV accession (e.g., `spectrum_MS_1000515_float` for all intensity values). The encoding varies wildly — base64+zlib, raw binary, HDF5 chunked, integer-scaled — but the semantic content is identical: **a buffer of numbers, a unit, an axis, and a precision**.

**Primitive 2: The Spectrum.** A collection of two or more SignalArrays sharing a common sampling domain. An MS spectrum pairs m/z and intensity arrays; an NMR 1D spectrum pairs chemical shift (ppm) and intensity arrays. NMR 2D spectra (COSY, HSQC) extend this to a matrix indexed by two frequency axes. Ion mobility MS adds a third signal dimension (1/K₀). The abstract `Spectrum` is dimensionally agnostic — it simply holds `n` named SignalArrays with `m` shared axes.

**Primitive 3: The AcquisitionRun.** A time-ordered sequence of Spectra from a single instrument session. In mzML, this is the `<run>` element containing `<spectrumList>` and `<chromatogramList>`. In Bruker NMR, it is the experiment directory (`<expno>/fid` or `<expno>/ser`). In Thermo RAW, it is the OLE2 container with scan-indexed binary streams. The Run carries instrument configuration, source file references, and acquisition timing metadata.

**Primitive 4: The CVAnnotation.** The `cvParam` pattern — `(ontologyRef, accession, name, value, unit)` — is the fundamental metadata atom shared across every HUPO-PSI format. mzML and nmrML use identical XML structures referencing different controlled vocabularies (PSI-MS CV vs. nmrCV). Both CVs trace their upper-level ontology to BFO (Basic Formal Ontology). This pattern maps directly to RDF triples and is the key enabler of semantic interoperability. The tools mzML2ISA and nmrML2ISA demonstrate that **23–46 metadata terms** can be automatically extracted from either format into ISA-Tab structures, confirming the shared metadata model.

**Primitive 5: The Identification.** Linking spectral features to chemical entities appears in mzIdentML (peptide-spectrum matches via `SpectrumIdentificationItem`), NMReDATA (chemical shift assignments mapped to atom indices in MOL structures), NMR-STAR (assigned peak lists referencing residue/atom identifiers), and mzTab-M (small molecule evidence linking MS features to compound identifications). The abstract pattern is always: **spectral coordinate → chemical entity → confidence score → evidence chain**.

**Primitive 6: The ProvenanceRecord.** Every format tracks processing history. mzML uses `<dataProcessing>` with ordered `<processingMethod>` steps. nmrML uses an identical structure. NMR-STAR records software and method metadata in save frames. The pattern is: **input → software × parameters → output**, directly mappable to the W3C PROV model's `Entity → wasGeneratedBy → Activity → used → Entity` chain.

## Container architectures: five philosophies, one clear winner

The formats examined use five distinct container strategies, and their relative merits illuminate what TTI-O should adopt.

**XML-based single files** (mzML, nmrML, mzIdentML, TraML) offer human readability, schema validation, and semantic validation through CvMapping rules. But XML imposes severe performance penalties: base64 encoding inflates binary data by **~33%**, compressed mzML files are **4–18× larger** than vendor equivalents, and random access requires either full sequential parsing or fragile byte-offset indices appended to file ends. The mzML schema has been frozen at version 1.1.0 since June 2009 — extensibility comes entirely through CV term additions rather than schema changes, which is architecturally elegant but means the format cannot evolve its data model.

**HDF5-based containers** (mzMLb, mz5, MZA, NeXus, Toffee) eliminate XML's performance problems while retaining self-describing metadata. mzMLb's key innovation is embedding complete mzML XML as an HDF5 character dataset while storing binary arrays as native HDF5 datasets with chunk-level compression. This eliminates base64 overhead entirely and enables cross-spectrum compression (impossible in mzML, where each spectrum is independently encoded). Benchmarks show mzMLb achieves file sizes **comparable to proprietary vendor formats** — an ~86% reduction with Numpress+zlib — while providing random access that is **orders of magnitude faster** than compressed mzML. The NeXus format from neutron/X-ray science demonstrates how to layer domain-specific semantics on HDF5 through `NX_class` attributes and application definitions, providing a direct model for a multi-omics container.

**SQLite-based metadata + separate binary** (Bruker TDF) is the most architecturally innovative vendor approach. The `analysis.tdf` SQLite database stores all metadata in relational tables queryable with standard SQL, while `analysis.tdf_bin` holds compressed spectral data. This separation enables metadata queries (e.g., "find all MS2 scans between 10–20 minutes with precursor m/z 500±1") without touching gigabytes of binary data. The SQL interface is vastly more powerful than XML parsing for structured queries. A 2025 benchmarking study (Kumler et al., *J. Proteome Research*) found that DuckDB completed common MS queries in under 1 second even on GB-scale data, **~10× faster than SQLite** for analytical workloads. Parquet files achieved the smallest file sizes at **30 MB** vs. mzMLb's 18 MB vs. mzML's 75 MB.

**Separated metadata + binary** (imzML) splits a single logical dataset into `.imzML` (XML metadata) and `.ibd` (raw binary) files linked by UUID. This achieves excellent binary efficiency — comparable to raw data sizes — and enables independent processing of metadata and data. But the dual-file architecture is fragile: file separation, orphaned binaries, and broken UUID links are persistent practical problems. The pattern is sound in principle but the UUID linking mechanism lacks integrity enforcement.

**Directory-based containers** (Bruker NMR `.d`, Waters `.raw`, Varian `.fid`) organize data as filesystem trees with separate files for parameters, raw data, processed data, and indices. Bruker NMR's hierarchy is particularly well-designed: `<expno>/fid` for raw data, `acqus` for frozen acquisition parameters in JCAMP-DX format, `pdata/<procno>/1r` for processed spectra, with 2D data using submatrix storage for efficient partial reads. The major weakness is portability — directories are not atomic units, and transferring them requires archiving (tar/zip), which destroys random access.

**The clear architectural winner is HDF5 with embedded semantic metadata**, as demonstrated by mzMLb. It combines: (a) the self-describing, hierarchically structured nature of HDF5; (b) preservation of full CV-based metadata from established standards; (c) native binary storage eliminating encoding overhead; (d) chunked, compressed datasets enabling random access and partial reads; and (e) broad language support (C/C++, Python, Java, R, Julia, MATLAB). The path toward cloud-native access exists through HSDS (HDF5 as a Service) and Zarr-compatible storage patterns.

## Mapping MPEG-G's architecture to spectral data

MPEG-G (ISO/IEC 23092), ratified in 2019–2021 through collaboration between ISO/IEC JTC 1/SC 29 (MPEG) and ISO TC 276 (Biotechnology), provides a remarkably complete template for what a multi-omics format standard needs. Its six-part structure — Transport and Storage, Coding, Metadata and APIs, Reference Software, Conformance, and Annotations — addresses every gap in the current MS/NMR format landscape.

**Dataset Group → Study.** MPEG-G's top-level container groups related datasets, analogous to how the ISA Investigation groups Studies. In TTI-O, a Dataset Group maps to a multi-experiment study (e.g., a clinical proteomics cohort or a multi-omics investigation combining MS and NMR data from the same biological samples). The ISA framework already defines this hierarchy — Investigation → Study → Assay — and TTI-O's container should be natively ISA-compatible.

**Dataset → AcquisitionRun.** MPEG-G's Dataset contains sequencing data from one experiment. In TTI-O, this maps to a complete LC-MS run, a single NMR experiment, or an MS imaging session. Each carries instrument configuration, processing history, and data class metadata.

**Access Unit → SpectrumGroup.** The Access Unit is MPEG-G's fundamental unit of random access — independently decodable, with a metadata header inspectable without decompressing the payload. For LC-MS data, the natural Access Unit is a **time window** of spectra (e.g., 1–2 seconds of acquisition, containing ~10–20 spectra at 10 Hz). For NMR 1D, it could be a single FID-spectrum pair. For 2D NMR, one indirect-dimension increment. For MS imaging, a **spatial tile** of 32×32 pixels. The key requirement is that each Access Unit's header is queryable in the compressed domain — MPEG-G achieves this through lightweight headers preceding each AU, and TTI-O should follow the same pattern.

**Descriptor Streams → SignalChannels.** MPEG-G's most powerful architectural concept is splitting genomic records into parallel typed streams — position descriptors, mismatch descriptors, quality values — each compressed independently with stream-specific entropy coding. The MS/NMR analog is immediately clear: **separate parallel streams for m/z values, intensity values, ion mobility values, and per-scan metadata**. This separation enables (a) selective access (read m/z without intensity), (b) optimized per-stream compression (m/z values have different statistical properties than intensities), and (c) columnar query support. mzMLb already partially implements this by storing all m/z values in one HDF5 dataset and all intensities in another. The 2025 mzsql benchmarking study confirms that columnar Parquet storage achieves the best compression ratios precisely because homogeneous columns compress far better than row-oriented mixed data.

**Data Classes → AcquisitionModes.** MPEG-G classifies genomic records into six classes (P, M, I, N, U, HM) by alignment type, enabling class-specific compression and filtering. An TTI-O analog would classify spectra by acquisition mode — **MS1 survey, MS2 DDA, MS2 DIA/SWATH, SRM/MRM, FID, processed 1D NMR, 2D NMR** — each with optimized compression strategies. MS1 spectra have different statistical properties than MS2 spectra (denser peaks, different dynamic range), justifying separate treatment.

**Protection containers → Multi-level encryption.** MPEG-G provides encryption and access control at every hierarchy level — Dataset Group, Dataset, Descriptor Stream, Access Unit — with owner-defined privacy rules. This directly addresses a critical gap: **no current MS or NMR format supports any form of encryption or access control**. Recent work by Bandeira et al. (*MCP*, 2021) and Rajczewski et al. (*MCP*, 2024) establishes that proteomics data enables re-identification through single amino acid variants, yet current repositories (PRIDE, MassIVE) lack controlled-access mechanisms. Clinical metabolomics faces similar GDPR/HIPAA considerations. TTI-O should implement MPEG-G's multi-level protection model, extended with proteomics-specific features like SAAV-containing spectrum anonymization.

**Part 3 APIs → Compressed-domain query interfaces.** MPEG-G Part 3 defines normative APIs for querying compressed data — retrieving genomic records by region, data class, or metadata properties without full decompression. No MS/NMR format provides equivalent APIs. The closest approach is mzMLb's HDF5-based partial reads (only chunks containing requested spectra are decompressed) and Bruker TDF's SQL-queryable metadata (filtering scans by properties without touching binary data). A hypothetical "mzsql" layer over TTI-O Access Units could enable queries like `SELECT mz, intensity FROM run WHERE rt BETWEEN 10 AND 12 AND ms_level = 2 AND precursor_mz BETWEEN 499 AND 501` — executed against compressed data with index-driven Access Unit selection.

## A concrete class hierarchy for TTI-O

Synthesizing the primitives and mapping analysis, the following object-oriented design captures the shared architecture while preserving domain-specific expressiveness. The design has three layers: interfaces defining capabilities, abstract base classes defining structure, and concrete classes for domain-specific specializations.

**Interfaces** encode the capabilities that MPEG-G provides but current formats lack:

`Indexable` supports random access by numeric index, key string, or range query — covering mzMLb's spectrum index, Thermo's scan-number lookup, and Bruker TDF's SQL-filtered access. `Streamable` supports progressive sequential access with seek capability — addressing the transport format requirement. `CVAnnotatable` provides the shared `cvParam` pattern used identically across mzML and nmrML. `Provenanceable` tracks processing history following W3C PROV. `Encryptable` provides MPEG-G-style multi-level protection with privacy rules and role-based access.

**Abstract base classes** define the shared structural skeleton:

`SignalArray` — the universal data atom. A typed numeric buffer (`float32`, `float64`, `int32`, `complex128`) with an `AxisDescriptor` (name, unit, value range, sampling mode), an `EncodingSpec` (precision, compression algorithm, byte order), and CV annotations. Every format's binary data reduces to this. `Spectrum` — a named collection of `SignalArray` instances sharing coordinate axes, with an index position, scan time, and optional precursor information. `AcquisitionRun` — a time-ordered, indexed, streamable sequence of `Spectrum` and `Chromatogram` objects with instrument configuration and provenance. `SpectralDataset` — the root container holding multiple runs, identifications, quantifications, and study-level metadata.

**Concrete classes** specialize for each domain:

`MassSpectrum` extends `Spectrum` with mandatory m/z and intensity arrays, optional ion mobility array, MS level, polarity, and scan window. `NMRSpectrum` extends `Spectrum` with chemical shift and intensity arrays, nucleus type, and spectrometer frequency. `NMR2DSpectrum` extends `Spectrum` with a 2D intensity matrix indexed by F1 and F2 axis descriptors, plus experiment type annotation. `FreeInductionDecay` extends `SignalArray` with real and imaginary components, dwell time, number of scans, and receiver gain. `Chromatogram` holds paired time-intensity arrays with chromatogram type annotation (TIC, XIC, SRM). `MSImage` extends `SpectralDataset` with spatial dimensions, pixel sizes, scan pattern, and a grid-indexed spectrum collection. `TransitionList` holds precursor-product transition definitions with retention time windows. `Identification` links spectrum references to chemical entities with scores and evidence chains. `Quantification` records abundance values with sample references and normalization metadata.

The design is intentionally **open for extension**, and the v0.11
release (M73, 2026-04) proved the extension point by shipping Raman
and IR as first-class modalities in all three reference
implementations. `RamanSpectrum` and `IRSpectrum` both extend the
same `Spectrum` base class with a shared `wavenumber` + `intensity`
pair plus technique-specific metadata (Raman: excitation wavelength,
laser power; IR: absorbance vs transmittance mode, resolution),
while `RamanImage` and `IRImage` reuse the `MSImage` rank-3-cube
pattern unchanged. No changes to the abstract base classes or
interfaces were needed — the hierarchy absorbed two new
spectroscopic techniques additively. UV-Vis, XRD, or other
techniques would follow the same template: subclass `Spectrum` (or
`MSImage` for hyperspectral) with a domain-specific x-axis and
metadata payload.

## What remains unsolved: security, streaming, cloud-native access, and standardization governance

**Security is the most glaring gap.** None of the 15+ formats analyzed include any form of encryption, digital signatures, or access control beyond SHA-1 checksums in indexedmzML and UUID linking in imzML. As clinical proteomics and metabolomics scale, this becomes untenable. MPEG-G's protection model — with selective encryption at every hierarchy level, owner-defined privacy rules in XML, and role-based key management — is directly transferable. The additional challenge for proteomics is that spectral data itself can be re-identifying through single amino acid variants, requiring spectral-level anonymization capabilities beyond what MPEG-G contemplated for genomics.

**Streaming and transport formats are entirely absent.** No MS or NMR format supports real-time streaming from instruments or progressive delivery over networks. Vendor instruments write proprietary streaming formats during acquisition, but these are never standardized. MPEG-G's dual File Format + Transport Format architecture, with normative bidirectional conversion, is the correct model. The natural streaming unit for LC-MS is a time-windowed Access Unit; for NMR, it is a single FID transient in a multi-scan acquisition.

**Cloud-native access is nascent.** The bioimaging community has Zarr (OME-Zarr) for cloud-optimized multi-resolution images. Genomics has cloud-optimized CRAM with htsget for selective retrieval. MS/NMR have nothing equivalent. HDF5's HSDS (HDF5 as a Service) provides a partial solution, and Parquet's footer-based metadata enables HTTP range requests for partial cloud reads. DuckDB can already query remote Parquet files directly. But no standard defines how to serve MS/NMR data from cloud object stores with selective access — a major gap as datasets grow beyond what local storage can handle.

**The standardization process itself needs reimagining.** HUPO-PSI's volunteer-driven, academic-led process has produced excellent, well-adopted standards (mzML has been stable for 17 years), but it lacks the resources for the architectural scope that TTI-O would require. MPEG's ISO process is more rigorous — with formal committee drafts, national body voting, conformance testing (Part 5), and reference software (Part 4) — but slower and more complex. The ideal approach is a **hybrid model**: community-developed technical specifications through existing PSI working groups and the Metabolomics Standards Initiative, elevated to formal ISO standardization for the interoperability layers (container format, transport protocol, API definitions, conformance testing). The RDA Working Group on "Creating a Multi-omics Metadata Schema Standard Reporting Matrix" is already attempting cross-omics harmonization. AnIML (ASTM E13.15) attempted a universal analytical data standard but stalled without sufficient adoption. The IUPAC FAIRSpec initiative is actively developing FAIR specifications for spectroscopic data. These efforts should converge rather than proceed independently.

**The ontology landscape is ready but imperfect.** BFO serves as the unifying upper-level ontology connecting PSI-MS CV, nmrCV, OBI, ChEBI, and UO. nmrCV explicitly imports terms from all of these using the MIREOT methodology. But analysis by ontology experts has found modeling inconsistencies — semantically similar classes not reused across ontologies, naming convention violations against OBO Foundry standards. For TTI-O, a reconciliation effort would harmonize the domain CVs under a shared TTI-O vocabulary layer, with BFO and OBI providing the upper-level framework and domain CVs (extended PSI-MS CV, nmrCV, ChEBI) providing technique-specific terms.

## Conclusion: the architectural case for TTI-O

The analysis reveals that the multi-omics format landscape has already converged on shared primitives — it simply hasn't formalized this convergence. The **SignalArray** with CV-annotated axis metadata is universal. The **cvParam** pattern for semantic annotation is shared between mzML and nmrML (and extensible to any spectroscopic technique). The **HDF5 container with embedded semantic metadata** (validated by mzMLb) provides the correct physical layer. The **ISA Investigation-Study-Assay hierarchy** provides the correct study-level metadata framework. And **MPEG-G provides the complete architectural template** for everything the current ecosystem lacks: hierarchical containers with Access Units for selective access, descriptor stream separation for columnar compression and partial reads, multi-level protection for clinical data privacy, streaming transport for real-time acquisition, normative APIs for compressed-domain querying, and formal conformance testing.

What TTI-O would add beyond a simple union of existing formats is the **architectural integration** that currently requires stitching together mzML + mzIdentML + mzTab + ISA-Tab + separate vendor files + ad hoc scripts. A single TTI-O container could hold LC-MS raw data, NMR FIDs, processed spectra, identifications, quantifications, and study metadata — all indexable, streamable, encryptable, and queryable through a standard API. The class hierarchy proposed here demonstrates this is not merely desirable but structurally natural: `MassSpectrum` and `NMRSpectrum` share more than 80% of their abstract interface. The Universal Spectrum Identifier (USI), ratified by PSI in 2021 and already covering **over 1 billion identifications from 3 billion+ spectra**, provides the persistent identifier infrastructure. The ISA framework provides the experimental metadata layer. The controlled vocabularies exist and interoperate through BFO.

The remaining challenge is not technical but organizational: coordinating the HUPO-PSI community, the Metabolomics Standards Initiative, the NMR standardization bodies (BMRB, NMReDATA initiative), the IUPAC FAIRSpec effort, and potentially ISO TC 276 (Biotechnology) into a coherent standardization process. MPEG-G required approximately 5 years from initial proposal to publication and bridged two ISO committees. A similar timeline and cross-committee structure would be appropriate for TTI-O — long enough to be rigorous, short enough to remain relevant in a field where data volumes are doubling annually and clinical applications demand privacy infrastructure that does not yet exist.
